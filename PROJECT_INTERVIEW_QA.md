# 医疗 KAG Multi-Agent 项目面试自问自答

> 使用方式：面试中先用“项目主线”讲 1-2 分钟，再根据面试官追问落到下面对应问题。回答时优先讲已实现内容，最后补一句可优化方向，避免把未来规划说成已经落地。

## 一、项目主线

**Q1：这个项目一句话怎么介绍？**

A：这是一个面向体检报告的医疗辅助评估 Multi-Agent 系统。用户输入自然语言体检文本后，系统先通过确定性 Workflow 完成指标解析、规则判定、Neo4j 图谱风险推理和证据检索，生成结构化诊断与建议；后续追问则由 Multi-Agent Supervisor 接管，结合个人健康画像、知识库检索和安全复核生成回答。

**Q2：它和普通 RAG 知识库问答最大的区别是什么？**

A：普通 RAG 多数是“问题 -> 检索 -> 回答”，核心在知识召回；这个项目核心是“体检数据 -> 医学规则 -> 图谱推理 -> 证据支撑 -> 个性化追问”。也就是说，LLM 不是直接判断健康风险，而是被放在确定性流水线和安全 Agent 后面，用于结构化结果解释、追问合成和表达优化。

**Q3：为什么这个项目要做成 Multi-Agent，而不是一个单 Agent？**

A：医疗体检场景有不同类型任务：初诊评估需要强约束，追问需要灵活上下文，知识问题需要检索，涉及用药又需要安全复核。把这些能力拆成分诊 Agent、评估 Agent、记忆 Agent、检索 Agent、合成 Agent、安全复核 Agent，能让每个角色边界清楚，也便于调试和扩展。

**Q4：如果面试官让你 2 分钟讲完整技术链路，你怎么讲？**

A：入口是 FastAPI 和 Streamlit。用户提交文本后，系统先判断是初诊还是追问。初诊进入 LangGraph 的 12 步 KAG Workflow：解析文本、归一化指标、按 JSON 规则识别异常状态、查 Neo4j 图谱得到疾病风险和干预路径、规划证据查询、从 Milvus/SQLite FTS5 检索证据、排序后生成结构化诊断。追问则进入 Multi-Agent Supervisor：先读取 PostgreSQL 中的事实、诊断、对话和摘要记忆；如果记忆足够回答就直接合成，如果需要医学知识就调用知识库检索工具；最后经过安全复核 Agent 检查用药和剂量风险后输出。

## 二、Agent 架构

**Q5：你项目里的 Agent 架构具体是怎么设计的？**

A：外层是 `MedicalAssessmentAgent`，内部持有一个 `MedicalMultiAgentSupervisor`。Supervisor 用 LangGraph StateGraph 编排 6 个角色节点：TriageAgent 负责分流初诊/追问，AssessmentAgent 调用确定性 Workflow，MemoryAgent 判断是否可用历史记忆，RetrievalAgent 调用医学知识库工具，SynthesisAgent 合成回答，SafetyReviewAgent 做医疗安全边界检查。

**Q6：为什么初诊不能直接交给 ReAct Agent？**

A：初诊涉及高风险判断，比如血压、eGFR、血糖这些阈值必须按医学规则执行，不能让 LLM 自由推理。项目里初诊强制走 Workflow，LLM 最多参与文本抽取和结果表达，核心风险判定由规则引擎、图谱和排序模块完成，这样可控性和可解释性更强。

**Q7：TriageAgent 如何判断初诊还是追问？**

A：当前实现用了一个保守的启发式判断：输入里出现体检相关关键词，并且同时出现数值或明确“请判断/体检报告/化验单”等评估意图，才判为初诊。只说“我的血压怎么样”这种一般判为追问，避免每次追问都重复跑完整体检流水线。

**Q8：这个 Multi-Agent 和 LangChain 的黑盒 Agent 有什么不同？**

A：这里没有直接把所有工具丢给一个 Agent Executor。每个 Agent 是显式图节点，状态字段、路由条件、事件输出都在代码里可见，比如 `needs_retrieval` 决定是否进入检索节点，`requires_safe_rewrite` 决定是否回到合成节点改写。好处是面试和生产排障时能解释“为什么走了这条路径”。

**Q9：项目中 ReAct 思路体现在哪里？**

A：追问路径里有明显的“观察上下文 -> 判断是否需要工具 -> 调用检索工具 -> 整合证据 -> 安全复核”的循环。MemoryAgent 相当于观察历史状态，RetrievalAgent 是工具调用，SynthesisAgent 根据观察和工具结果生成回答，SafetyReviewAgent 如果发现剂量或药名风险，会路由回 SynthesisAgent 触发安全改写。

**Q10：如果安全复核一直要求改写，会不会死循环？**

A：不会。状态里有 `safety_revision_count`，当前只允许首次发现剂量/疑似药物风险时触发改写。改写后再次进入安全复核，如果没有新的风险就结束；如果仍然有提示，也会以安全说明形式输出，避免无限循环。

## 三、确定性 Workflow

**Q11：LangGraph Workflow 里有哪些步骤？**

A：Workflow 有 12 个步骤：解析输入、校验结构、指标标准化、异常规则判定、图谱风险检索、干预路径扩展、证据查询规划、证据检索、风险排序、主诊断生成、次级建议生成、封装响应。所有步骤由同一份 `_step_definitions()` 维护，同步、异步和流式事件共用，避免多套流程不一致。

**Q12：为什么 Workflow 还要接 LangGraph？顺序执行不也可以吗？**

A：顺序执行可以完成原型，但 LangGraph 更适合表达状态机和条件边。项目里用它把步骤节点化，后续可以自然扩展条件短路、异步节点、流式事件和可观测日志。尤其是医疗场景需要追踪每一步中间状态，LangGraph 的状态对象更清楚。

**Q13：健康路径为什么做条件短路？**

A：如果规则引擎没有识别到异常状态，就没有必要访问图谱和证据库。代码里对图谱检索、证据规划、证据检索、排序等步骤做了 skip，直接生成健康结论。这样能降低正常体检场景延迟，也减少外部组件不可用对健康路径的影响。

**Q14：异步优化做在了哪里？**

A：在异步 Workflow 中，证据检索和干预路径扩展是并行执行的。因为它们都依赖风险候选，但彼此没有依赖关系，所以用 `asyncio.gather` 同时调用 evidence store 和 graph store，减少异常路径的 wall-clock 时间。

## 四、输入解析、规则与诊断

**Q15：体检文本是怎么解析成结构化数据的？**

A：输入解析支持三类入口：已经结构化的 JSON、字符串中的 JSON、自然语言文本。自然语言优先走 LLM structured output 抽取；如果模型不可用或失败，就回退到正则解析血压、年龄、性别、病史、用药史以及常见指标。

**Q16：指标归一化解决什么问题？**

A：同一个指标可能有不同写法，比如 LDL、低密度脂蛋白、LDL-C；单位也可能不同，比如血糖 mg/dL 和 mmol/L、肌酐 mg/dL 和 umol/L。`IndicatorNormalizer` 把别名映射到统一 code，并做单位换算，保证后续规则和图谱都用稳定字段。

**Q17：规则引擎为什么用 JSON 配置？**

A：医学阈值变化或新增指标时，不应该频繁改业务代码。项目把单指标规则和组合规则放在 `medical_rules.json`，代码只负责通用的条件求值，支持 `gte/gt/lte/lt/eq/in/not_in` 和嵌套 AND/OR。这样扩展新指标时主要改配置。

**Q18：组合规则是怎么实现的？**

A：规则配置里可以写多个条件，也可以嵌套 conditions。比如某些风险不只看一个指标，还要结合年龄、性别或另一个化验值。规则引擎会先构建 `values_by_code`，再递归判断条件树，匹配后生成一个 `DetectedState`。

**Q19：主诊断和建议是怎么生成的？**

A：Workflow 先通过规则得到异常状态，再由图谱得到风险候选和干预路径，然后用 `MedicalRiskRanker` 综合图谱支持和证据支持排序。`DiagnosisFormatter` 最后生成健康状态、紧急程度、潜在风险、建议科室、复查项目、生活方式建议和 `human_review_required`。

## 五、Neo4j KAG 图谱

**Q20：为什么要引入 Neo4j 图谱？**

A：体检评估不是单纯文本相似度问题。比如 eGFR 降低和肌酐升高都可能指向慢性肾病风险，风险还要连接到复查项目、科室和禁忌。Neo4j 可以把“异常状态 -> 风险 -> 疾病 -> 建议”建成显式关系，让系统具备路径推理和可解释性。

**Q21：图谱里建了哪些核心关系？**

A：核心链路是异常状态到疾病风险，再到疾病，再到干预、复查、科室、禁忌。Workflow 根据规则输出的 state_code 查询风险候选，再根据 disease_code 扩展干预路径。返回结果里保留 `GraphPath`，用于解释判断来源。

**Q22：为什么还要有 InMemoryGraphStore？**

A：这是为了本地开发、测试和降级。Neo4j 不可用时系统可以自动回退到内存图谱，不影响核心演示和单元测试。生产上可以用 Neo4j，测试环境用 InMemory，接口保持一致。

**Q23：图谱和检索是怎么结合的？**

A：图谱输出的 risk_code 和 disease_code 会作为 node_codes 传给证据检索。检索排序时会计算证据 chunk 的 linked_node_codes 与当前风险节点的重叠度，这个 graph overlap score 会参与最终排序，让证据更贴近当前风险路径。

**Q24：如果图谱没有命中风险怎么办？**

A：代码里会跳过干预路径扩展，但证据检索仍可兜底。这个设计避免“图谱未覆盖就完全无结果”，同时也能把低置信场景标记为需要人工复核。

## 六、检索增强

**Q25：证据检索链路是怎样的？**

A：检索先由 `EvidenceQueryPlanner` 根据体检结果、异常状态和风险候选构造多条查询；然后 evidence store 做向量召回、词汇检索、Rerank、图谱重叠度和来源权威度加权，最后用 MMR 做多样性选择，返回 top_k 证据 chunk。

**Q26：Milvus 在项目里承担什么职责？**

A：Milvus 用于证据 chunk 的 dense 向量检索。优化后 Milvus 路径会复用 Milvus search 返回的 distance/score，不再把候选拿回来后在 Python 里重新对所有向量做点积，减少重复计算。

**Q27：为什么词汇检索用 SQLite FTS5？**

A：项目定位是可本地部署的原型系统，不想引入 Elasticsearch 这类额外服务。SQLite FTS5 零外部依赖、轻量、持久化，适合医学术语、指标名、指南标题这种关键词召回。中文分词不足时，代码还保留了 fallback LIKE 思路。

**Q28：Rerank 放在检索链路里的作用是什么？**

A：dense 和 lexical 召回更多解决“能不能召回”，Rerank 更关注“谁排前面”。项目会把前置候选交给远程 Rerank 模型重排，再和 dense、lexical、图谱重叠度、来源权威度一起融合，提升 top 结果的语义相关性。

**Q29：MMR 为什么需要？**

A：医学指南 chunk 容易多个片段内容高度相似。如果只按相关性排序，top5 可能都是同一段附近的重复证据。MMR 在相关性和多样性之间折中，尽量让最终证据覆盖诊断依据、干预建议、复查方向等不同方面。

**Q30：检索这块还有什么可优化？**

A：可以做三类优化。第一，建立更严格的人工标注评测集，按疾病/指标/建议类型分桶评估。第二，增强中文医学词汇索引，比如接入医学词典、同义词扩展和拼写归一化。第三，进一步把图谱路径作为 query rewrite 输入，让检索不仅知道疾病名，还知道当前问题处于“诊断/复查/干预/禁忌”哪个意图。

## 七、记忆与个体健康画像

**Q31：项目里的记忆不是普通聊天记录，具体体现在哪里？**

A：项目把记忆拆成结构化健康画像，而不是只保存 role/content。PostgreSQL 中有会话表、对话记忆表、用户事实记忆表和诊断记忆表。事实记忆保存年龄、性别、指标、病史、用药、过敏；诊断记忆保存每次结构化评估结果；对话记忆和摘要记忆用于追问上下文。

**Q32：事实记忆如何处理新旧冲突？**

A：新体检报告入库时，系统会读取当前会话已有事实，用 `(fact_group, fact_key)` 做对比。如果同一个指标数值发生变化，比如收缩压从 176 更新为 152，会生成“已更新事实”的系统消息，而不是静默覆盖。这对体检趋势解释很重要。

**Q33：诊断记忆如何支持趋势对比？**

A：每次结构化评估会写入 `DiagnosticMemory`，有 `version_no` 和 `is_current`。新增诊断时会把上一版标成非 current，然后写入新版。构建上下文时会取最近两版，生成“最新 vN vs 上次 vN-1”的趋势摘要。

**Q34：摘要记忆为什么设置触发阈值？**

A：每次追问都调用 LLM 摘要会增加延迟和成本。项目里会累计新增字符数，超过配置阈值才允许调用摘要模型；未达到阈值时使用确定性摘要。这是成本、实时性和上下文质量之间的折中。

**Q35：记忆这块有哪些可优化？**

A：当前事实记忆是结构化存储和确定性筛选，尚未真正做事实记忆向量检索。后续可以给 UserFactMemory 增加 embedding 字段或独立向量 collection，根据当前追问召回最相关事实，而不是只截取固定数量事实；还可以对指标做时间序列建模，支持“近三次变化趋势”和异常持续时间分析。

## 八、安全与医疗边界

**Q36：医疗安全边界是怎么做的？**

A：项目不是只在回答末尾加免责声明，而是分层控制。规则引擎负责确定性阈值，Workflow 强制初诊走完整链路，图谱限制风险推理方向，检索提供证据支撑，SafetyReviewAgent 检查用药、剂量、疑似药名和高风险结果，必要时触发安全改写。

**Q37：SafetyReviewAgent 具体拦什么？**

A：它会检查用户问题是否涉及用药调整，回答里是否出现具体剂量或服药频次，比如 mg、片、次/日，也会用正则识别疑似药物词，如沙坦、他汀、洛尔、地平等。如果发现风险，首次会路由回 SynthesisAgent 生成非处方化的安全回答。

**Q38：`human_review_required` 什么时候会触发？**

A：诊断格式化阶段会根据风险等级和复杂程度设置。比如存在高风险疾病候选时，次级建议里会标记需要人工复核。SafetyReviewAgent 看到结构化结果里这个字段后，也会在事件中附加“高风险或复杂结果需要医生复核”。

**Q39：知识库防污染做到了什么程度？**

A：当前已实现文件类型白名单、空文件拒绝、SHA-256 去重、文本抽取、chunk 入库和图谱节点推断。它能避免重复文件和明显非法格式污染知识库。可优化方向是增加更强的医学相关性门控，比如关键词密度、图谱节点覆盖率、来源权威度和低分拒绝策略，而不仅是推断节点后入库。

**Q40：如果 LLM 抽取错了怎么办？**

A：项目设计上不完全依赖 LLM。抽取失败会回退正则；后续还有指标归一化、规则引擎和缺失字段 warning。可优化方向是加入抽取置信度、原文 span 对齐、异常值二次确认，以及对关键指标做 deterministic parser 优先。

## 九、API、流式与前端

**Q41：FastAPI 提供了哪些核心接口？**

A：核心接口分四类：体检解析 `/exam/parse`，体检评估 `/exam/assess`，Agent 对话 `/agent/chat` 和 `/agent/chat/stream`，知识库上传/重建/列表，会话创建/查询/删除，以及 `/runtime/status` 组件状态接口。

**Q42：SSE 流式回答是怎么实现的？**

A：`/agent/chat/stream` 返回 `StreamingResponse`，内部异步遍历 `medical_agent.stream_assess_async()`。前端会收到 meta、agent_decision、tool_call、tool_result、step、content、result、done 等事件。这样用户能看到处理过程，而不是等最终答案一次性返回。

**Q43：为什么要把处理过程事件化？**

A：医疗评估需要可解释性。事件流能展示系统当前在解析指标、查图谱、检索证据、合成回答还是安全复核。出了问题也能定位是解析失败、图谱未命中、检索为空，还是安全复核触发改写。

**Q44：Streamlit 前端在项目里承担什么角色？**

A：Streamlit 是工作台前端，用于提交体检文本、展示流式过程、显示结构化结果、查看会话历史、上传知识库文档和查看运行时状态。它不是核心算法部分，但让项目具备完整可演示闭环。

## 十、知识库治理与数据后端

**Q45：文档上传链路怎么设计的？**

A：上传接口先校验文件后缀和内容，再计算 SHA-256 去重，创建上传任务并交给 FastAPI BackgroundTasks 后台处理。后台解析 txt/md/html/json/pdf，切 chunk，推断关联图谱节点，写入 registry，并增量加入 evidence store 和 graph store。

**Q46：为什么上传任务要后台化？**

A：PDF 解析、chunk、embedding 和索引写入可能比较耗时。如果同步阻塞上传请求，前端体验差，也容易超时。后台任务能让接口快速返回 job_id，前端再轮询任务状态。

**Q47：PostgreSQL 存哪些数据？**

A：PostgreSQL 主要存会话和记忆：`chat_sessions` 存会话标题、摘要和更新时间；`conversation_memories` 存原始对话和摘要；`user_fact_memories` 存结构化事实；`diagnostic_memories` 存版本化诊断结果。知识库元数据当前使用 JSON registry 文件。

**Q48：Neo4j、Milvus、PostgreSQL 三者职责怎么区分？**

A：Neo4j 存医学关系和推理路径，回答“异常状态会关联哪些风险和建议”；Milvus 存证据 chunk 的向量索引，回答“哪些资料片段语义相关”；PostgreSQL 存用户会话和个人健康画像，回答“这个用户之前是什么状态”。

## 十一、降级、可观测与测试

**Q49：外部组件不可用怎么办？**

A：GraphStore 和 EvidenceStore 都有 remote/in-memory 双后端。Neo4j 或 Milvus 初始化失败、ping 失败时，会回退到内存实现，并在 runtime status 中暴露 backend、mode、fallback_reason。模型层也有轻量 hash embedding 作为兜底。

**Q50：系统可观测性体现在哪里？**

A：第一是流式事件，面向用户展示 Agent 决策和 Workflow 步骤。第二是结构化日志，每个 Workflow step 记录耗时、指标数、异常状态数、风险候选数和证据数。第三是 `/medical/runtime/status`，展示图谱、证据库、数据库、embedding、extractor、reranker 等组件状态和延迟。

**Q51：测试覆盖了哪些内容？**

A：测试覆盖输入解析、指标归一化、规则配置、KAG Workflow、证据检索、知识库构建、Agent 工具、Multi-Agent 路由、会话记忆、异步一致性和运行时状态。测试里大量使用 InMemoryGraphStore/InMemoryEvidenceStore，保证不依赖外部 Neo4j/Milvus 也能跑核心逻辑。

**Q52：你如何证明这个项目不是只能演示？**

A：它有完整后端 API、前端工作台、文档上传、会话记忆、组件健康检查、外部服务降级和回归测试。虽然还不是生产医疗系统，但工程结构已经按可部署、可扩展、可观测的方向拆分了模块。

## 十二、技术取舍与可优化方向

**Q53：项目里你最重要的技术取舍是什么？**

A：最重要的是把“医疗风险判断”从 LLM 中拿出来，放到规则、图谱和 Workflow 中；把 LLM 用在抽取、表达和追问合成上。这样牺牲了一些自由度，但换来了安全边界和可解释性，更适合体检辅助评估场景。

**Q54：如果要进一步生产化，优先优化什么？**

A：第一，补充真实医学标注集，按疾病和任务类型评估解析、图谱推理和证据检索。第二，增强事实记忆向量召回和时间序列趋势分析。第三，完善知识库医学相关性门控和来源权威度体系。第四，加入用户权限、审计日志、敏感信息脱敏和医生复核工作流。

**Q55：如果数据规模扩大到几十万 chunk，检索层怎么改？**

A：Milvus 侧要优化 collection schema、分区、HNSW/IVF 参数和批量写入；FTS5 对中等规模可用，规模更大或多租户复杂查询时可以迁移到 PostgreSQL tsvector 或 Elasticsearch/OpenSearch；同时要做候选截断、rerank 批量化和缓存热门 query。

**Q56：如果要支持更多疾病领域，怎么扩展？**

A：优先扩展三层：规则配置增加指标和阈值，图谱增加状态、风险、疾病和干预关系，知识库上传对应指南。代码层主要复用 parser、normalizer、rule engine、graph store 和 evidence store，不需要改 Agent 主流程。

**Q57：如果面试官质疑“这算不算医疗诊断系统”，怎么回答？**

A：我会明确说它是体检辅助评估和健康风险提示系统，不替代医生诊断。系统输出里有人工复核标记，涉及用药和高风险场景会触发安全复核，核心目标是把体检数据结构化、风险路径解释清楚、给出复查和就医方向。

**Q58：项目目前最大的短板是什么？**

A：第一，医学知识和规则仍是原型规模，覆盖面有限。第二，事实记忆还没有真正做语义向量召回。第三，文档入库相关性门控还可以更严格。第四，评测数据偏离线弱标注，生产前需要真实人工标注集和医生评审。

**Q59：如果被问到你个人贡献，怎么说？**

A：可以按模块回答：我负责整体 Agent 架构设计，把初诊确定性 Workflow 和追问 Multi-Agent 分开；实现输入解析、规则引擎、图谱推理、证据检索融合、四层记忆和安全复核；同时补齐 FastAPI/Streamlit 闭环、组件降级、运行时状态和测试。

**Q60：项目最能体现 Agent 开发多样性的点是什么？**

A：第一个层面是确定性 Agent：AssessmentAgent 不追求自由对话，而是严格执行医学 Workflow。第二个层面是协作式 Agent：分诊、记忆、检索、合成、安全复核各司其职。第三个层面是工具型 Agent：RetrievalAgent 通过 StructuredTool 调用知识库。它展示的是 Agent 不只是聊天机器人，也可以是带强约束的业务流程编排器。

## 十三、面试收束话术

**Q61：最后如何总结这个项目的技术亮点？**

A：这个项目的亮点不是单点模型调用，而是把医疗体检场景拆成了确定性评估、图谱推理、证据检索、个体记忆和安全复核五个层次。通过 Multi-Agent Supervisor 把这些层次组织起来，既能输出可解释的结构化评估，也能在多轮追问中利用个人健康画像进行个性化回答。

**Q62：如何自然引出可优化方向？**

A：可以说：“当前版本已经完成了从体检文本到结构化评估、多轮追问和安全复核的闭环。下一步我会优先做三件事：真实标注集评测、事实记忆向量化和知识库入库强门控。这三点分别对应效果可信度、个性化能力和知识库质量。”
