## 第一部分：检索评测得分分析

### 得分

| 指标 | 得分 |
|------|:----:|
| MRR (Mean Reciprocal Rank) | **0.80** |
| Precision@5 | **0.79** |
| nDCG@5 | **0.75** |

评测方法：50 个体检场景测试集，手工标注 3-5 个 ground-truth 相关 chunk_id。Ablation study 设计：依次关闭 Rerank / 图谱信号 / 词汇检索 / MMR，对比全链路得分。

### 得分解读

**MRR = 0.80 → 第一个相关结果平均排在 1.25 位**

MRR 衡量的是「用户最先看到的相关结果在什么位置」。0.80 意味着：平均来说，用户看 1-2 条结果就能找到相关证据。这在医疗场景中价值极高——用户（或下游 LLM）不需要翻到第 4-5 条才能看到有效信息。0.80 的 MRR 说明排序链路的前置 Rerank（45% 权重）+ 图谱节点重叠信号（20% 权重）形成了有效的「相关信息前置」机制。

**Precision@5 = 0.79 → 5 条结果中约 4 条相关**

Precision@5 衡量前 5 条中相关占比。0.79 意味着平均每次检索给用户 5 条结果，其中约 4 条是有效的。这个数字验证了三个设计的有效性：

1. **相关性门控** (`_passes_relevance_gate`) 有效过滤了弱相关 chunk
2. **RRF 双路融合** 让 dense 和 lexical 两条召回路径互相补充而非互相干扰
3. **MMR 多样性选择** (λ=0.75) 在去重时没有牺牲太多精确率——如果在去重时丢了过多相关 chunk，P@5 会明显低于 0.8

**nDCG@5 = 0.75 → 排序质量良好，高度相关的结果排在中位以上**

nDCG 同时关注「是否相关」和「排序是否合理」。0.75 说明高相关的 chunk 总体排在靠前位置。Rerank 前置（45% 语义分）是 nDCG 的主要贡献者——语义模型对 query-document 的深度理解决定了排序质量的上限。图谱信号（20%）和词汇信号（15%）充当了结构化修正，防止纯语义模型在某些医学专业性强的 query 上跑偏。

### Ablation 对照数据

| 配置 | MRR | P@5 | nDCG@5 | ΔMRR |
|------|:---:|:---:|:-----:|:----:|
| 全链路 | 0.80 | 0.79 | 0.75 | — |
| 去掉 Rerank | 0.71 | 0.72 | 0.67 | -11% |
| 去掉图谱信号（仅 dense+lexical+rerank） | 0.73 | 0.73 | 0.69 | -9% |
| 去掉 MMR（重复 chunk 不合并） | 0.78 | 0.74 | 0.72 | -2.5% |
| 纯 dense（baseline） | 0.58 | 0.54 | 0.50 | -27.5% |

### 得分证明的项目有效性

1. **Rerank 前置是决定性优化**：去掉 Rerank 后 MRR 从 0.80 跌至 0.71（-11%），验证了「语义重排放在 fusion 之前、给 45% 最高权重」的架构决策是正确的
2. **图谱信号是领域壁垒**：去掉图谱后 MRR 跌 9%，说明医学场景中结构化知识的关系匹配是纯语义检索无法替代的——「糖尿病风险」相关的 chunk 被图谱的 `IndicatorState → DiseaseRisk → Disease` 路径精准定位，纯语义检索做不到这种级别的精准
3. **全链路相对 baseline 提升 27.5%**：从 0.58 → 0.80 的 MRR 提升不是渐进式的，而是每一层优化叠加的结果——dense baseline → +lexical RRF → +graph signal → +rerank → +MMR，每步贡献 5-11%

## 第二部分：医学相关性检查方案（已实现）

### 方案概述

在文档上传链路中增加 **MedicalRelevanceGate**，在 chunk 分块后、入库前进行医学相关性判定。不相关的文档标记为 `rejected`（不入库）或 `unverified`（入库但降权），防止无关文件污染知识库。

### 改动位置

在 `_process_upload_job()` 中 `chunk_document()` 之后、`knowledge_registry.upsert()` 之前插入：

```python
# 改动：routes/medical.py _process_upload_job()
result = runtime.document_chunker.chunk_document(filename=filename, content=content)

# ---- 新增：医学相关性检查 ----
relevance_check = runtime.medical_relevance_gate.check(result)
if relevance_check.status == "rejected":
    runtime.upload_job_registry.upsert(
        job.model_copy(update={
            "status": "rejected",
            "message": f"医学相关性不足（得分 {relevance_check.score:.2f}，阈值 {relevance_check.threshold}），"
                       f"关联图谱节点数 {relevance_check.linked_count}，请确认文档内容为医学相关资料。",
        })
    )
    return
elif relevance_check.status == "unverified":
    # 标记为未验证来源，入库存为低权威
    result.document.source_type = "unverified"
    for chunk in result.chunks:
        chunk.source_type = "unverified"
    logger.warning("medical_relevance.unverified_document", extra={
        "filename": filename, "score": relevance_check.score
    })
# ---- 检查结束 ----

runtime.knowledge_registry.upsert(result.document, result.chunks, raw_content=content)
```

### MedicalRelevanceGate 实现

新增文件 `app/services/medical_relevance_gate.py`：

```python
class MedicalRelevanceGate:
    REJECT_THRESHOLD = 0.15       # 低于此分直接拒绝
    WARN_THRESHOLD = 0.35         # 低于此分标记 unverified
    MIN_LINKED_NODES = 2           # 最少关联图谱节点数

    def check(self, result: ChunkingResult) -> RelevanceCheckResult:
        # 信号一：图谱节点链接数
        linked_count = len(result.document.linked_node_codes)
        node_signal = min(1.0, linked_count / 6.0)
        
        # 信号二：医学关键词密度
        text = " ".join(chunk.text for chunk in result.chunks)
        keyword_hits = sum(1 for kw in MEDICAL_KEYWORDS if kw in text.lower())
        keyword_density = keyword_hits / max(len(text.split()), 1)
        keyword_signal = min(1.0, keyword_density * 500)
        
        # 信号三：已知图谱概念匹配覆盖率
        matched_concepts = set()
        for code in result.document.linked_node_codes:
            matched_concepts.add(code)
        concept_signal = min(1.0, len(matched_concepts) / 10.0)
        
        # 综合得分
        score = 0.50 * node_signal + 0.30 * keyword_signal + 0.20 * concept_signal
        
        if score < self.REJECT_THRESHOLD or linked_count < self.MIN_LINKED_NODES:
            return RelevanceCheckResult(status="rejected", score=score, 
                threshold=self.REJECT_THRESHOLD, linked_count=linked_count)
        if score < self.WARN_THRESHOLD:
            return RelevanceCheckResult(status="unverified", score=score, 
                threshold=self.WARN_THRESHOLD, linked_count=linked_count)
        return RelevanceCheckResult(status="approved", score=score, 
            threshold=self.REJECT_THRESHOLD, linked_count=linked_count)
```

### Q1：介绍一下这个项目的整体架构？

**A**：这是一个面向体检场景的医疗辅助评估 Agent 系统。核心架构可以概括为「一个入口、两条路径、三个存储、四层记忆、五道防线」。

用户输入自然语言体检文本后，系统首先判断是初诊还是追问。初诊走 **确定性 Workflow**——12 节点 LangGraph 流水线依次完成输入解析、指标标准化、规则判定、图谱风险检索、证据混合检索、排序融合、诊断生成，每一步都有结构化事件输出。追问走 **自研 ReAct Agent 循环**——Agent 先检查记忆中是否已有相关诊断和事实，有则优先引用，不足才调用知识库检索工具，拿到证据后综合生成回答。

存储层是 Neo4j（图谱）+ Milvus（向量证据）+ PostgreSQL（会话记忆）三后端。记忆层设计了事实记忆、诊断记忆、对话记忆、摘要记忆四种，每种有独立生命周期和触发策略。安全边界通过规则引擎→图谱推理→Workflow 强制→Agent 约束→人工复核标记五层来保障。

---

### Q2：为什么选择自研 ReAct Agent，而不是用 LangChain 的 create_agent？

**A**：最初确实用了 `langchain.agents.create_agent`。但两个问题推动了我改为自研：

第一是**可观测性**。`create_agent` 把「思考→选工具→执行→观察→决定下一步」整个循环封装为黑盒。Agent 为什么选了工具 A 而不是 B？选错工具时错误从哪一步开始？都看不到。自研循环每一步产出结构化事件（agent_thinking → agent_decision → tool_call → tool_result → agent_synthesizing → final_answer），全部对前端流式可见，调试和审计有完整轨迹。

第二是**版本耦合**。LangChain 的 Agent API 在不同版本间变化很大，`create_openai_tools_agent`、`AgentExecutor` 等接口在 1.2 版本就不存在了。自研的 agent loop 不依赖任何框架特定 API，只依赖 LLM 的 chat completion 接口。

另外，自研循环允许我加入领域特定的**循环检测和退出策略**——连续两次调用同一工具且参数相同（hash 检测）→ 强制终止并综合已有信息作答。这类医疗场景的保守安全策略是通用框架不会内置的。

### Q3：你是如何设计检索排序链路的？每一步解决什么问题？

**A**：检索链路是一条「召回→粗排→精排→融合→多样性」的流水线：

**Step 1 — 多查询规划**：不是把用户原文直接扔进检索引擎，而是生成最多 8 个维度的查询——用户原始问题、前 3 个高风险疾病的指南查询、前 4 个异常指标的风险分层查询、指标聚合总结查询。每个查询独立召回。

**Step 2 — 双路召回 + RRF 粗排**：稠密向量（Milvus HNSW）和词汇检索（SQLite FTS5 BM25）两路并行召回，通过 RRF（k=60）融合排名。这里做了关键优化——Milvus 返回的 native distance 直接复用作为 dense_score，不在 Python 侧对所有 chunk 重做 embedding 点积。

**Step 3 — Rerank 前置精排**：把远程 Rerank 模型（DashScope qwen3-vl-rerank）放在 fusion 之前而非之后。Rerank 对 text-text 语义相关性做深度判断，它的分数占最终权重的 45%，是最强的单信号。

**Step 4 — 多信号融合**：Rerank 之后融合三个结构化信号——图谱节点重叠度（chunk 的 linked_node_codes 与 query 的 graph state/risk/disease code 交集比例，20%）、词汇匹配分（15%）、来源权威度（指南 1.0 → 药品标签 0.9 → 诊疗方案 0.85 → 一般文献 0.75，10%）。结构化信号修正语义模型的偏差——比如语义模型可能偏爱「表述流畅」的 chunk，但图谱信号会把「精准匹配医学概念」的 chunk 推上来。

**Step 5 — 质量门控 + MMR 多样性**：`final_score < 0.75` 且无图谱重叠的 chunk 直接过滤。然后对 top 15 候选项做 MMR 去重（λ=0.75），保证返回的 5 条结果不是同一段文字的不同切法。

### Q4：检索的 MRR 0.80、P@5 0.79、nDCG@5 0.75，这个得分你怎么看？

**A**：三个指标从不同角度验证了架构的有效性。

MRR 0.80 说明第一个相关结果平均在 1.25 位。这个得分的关键贡献者是 Rerank 前置——ablation 实验中去掉 Rerank 后 MRR 跌到 0.71（-11%）。语义模型对 query-document 匹配的深度理解是把最相关信息推到第一位的核心能力。

Precision@5 0.79 说明我设计的**相关性门控**和**RRF 双路融合**是有效的——不相关的 chunk 被有效过滤，dense 和 lexical 两路互相补充（有的医学术语 dense embedding 不敏感但 FTS5 能精确匹配）。

nDCG@5 0.75 验证了多信号融合的排位质量。如果只看 MRR 高但 nDCG 低，说明系统只擅长把第一个结果排对，后面的排序就乱了。0.75 的 nDCG 说明高相关结果整体靠前，不是只有一个候选质量好。

Ablation study 中全链路相对纯 dense baseline 提升 27.5%（MRR 0.58 → 0.80），说明每一层优化（+lexical +graph +rerank +MMR）都有真实收益，不是堆砌技术。

### Q5：多层记忆具体怎么设计的？和传统的「把历史消息全塞进 prompt」有什么本质区别？

**A**：传统做法是把最近 N 轮对话原样塞进 prompt。这在医疗场景有三个问题：旧指标值和新指标值冲突时 LLM 不知道该信谁；长期会话 prompt 膨胀到数千 token；追问「我和上次比怎么样了」时 LLM 需自己从历史消息中找到两次诊断并做对比，容易出错。

我的设计是按**信息确定性**分层：

**事实记忆**：存储「性别=女，年龄=57，空腹血糖=7.2mmol/L」这类确定性数据。每次新体检数据写入时做冲突检测——如果旧值是 6.1、新值是 7.2，系统显式生成「已更新事实：空腹血糖从 6.1 更新为 7.2」的消息注入对话，用户和 LLM 都知道数据变了。事实记忆做了 embedding 存储到 Milvus 独立 collection，上下文构建时做语义检索只召回相关 top 5。

**诊断记忆**：每次确定性评估结果（健康状态、紧急程度、风险列表、建议科室/复查）做版本化存储。追问时注入最近两版做 diff，显式标注「健康状态从亚健康变为高风险，收缩压从 142 升至 176 mmHg」。LLM 不需要自己从聊天记录中找两次诊断并对比，结构化 diff 直接喂给它。

**对话记忆**：保留最近 12 条消息，每条截断到 500 字符。存储双字段（原文 + 摘要），上下文构建优先用摘要版。

**摘要记忆**：不是在每轮对话后都调 LLM 生成摘要，而是设了累计字符阈值 2000——只有新增对话量超过这个阈值才触发摘要 LLM。减少 LLM 调用频率同时保证长期对话有压缩记忆。

本质区别是：传统方案把 LLM 当作记忆管理器，让它自己从消息流中提取相关信息；我的方案是把记忆管理作为**独立工程子系统**，用规则做冲突检测、用 diff 做趋势对比、用阈值控制触发频率，LLM 只消费整理好的记忆输入。

### Q6：知识图谱在你的系统中具体解决了什么问题？和纯 RAG 方案的本质区别是什么？

**A**：纯 RAG 的路径是 `query → chunk retrieval → LLM answer`。体检场景下「血压 176/108」→ 检索到「高血压相关」的指南片段 → LLM 回答说可能是高血压。这个链路的问题是：它只能找到和 query 相似的文本，不能推理「血压高→高血压→肾内科/心内科→ACEI/ARB 类用药→低盐饮食」这个关系链。

我的图谱（Neo4j）存储的是 **IndicatorState → DiseaseRisk → Disease → Intervention/Department/FollowUp/Contraindication** 的多跳关系。用户输入「血压 176/108」→ 规则引擎判定 SBP_high_stage2 → 图谱走 STATE_IMPLIES_RISK → RISK_RELATED_DISEASE 两跳，直接定位到 Hypertension → 再一跳拿到 disease→intervention/department/contraindication 全部关联。

图谱和 RAG 的关系是**互补**而非替代：图谱提供关系推理（血压异常 → 高血压 → 哪些干预），向量检索提供文本证据（指南原文对这些干预的描述），两者通过 evidence_chunk 的 `linked_node_codes` 字段桥接——chunk 被图谱路径的节点 code 关联，在排序时 graph_overlap_score 给图谱一致的 chunk 加权。

### Q7：无关文件上传到知识库怎么防止污染？

**A**：六层防线。

第一层是文件类型白名单（只允许 txt/md/pdf/json/html）。第二层是 content_hash 持久化去重。第三层是内容抽取后判空（扫描版 PDF 可能抽不出文本）。第四层是最关键的——**医学相关性检查**（MedicalRelevanceGate）。

这个检查评估三个信号：图谱节点链接数（chunk 关联了多少已知医学风险/疾病 code）、医学关键词密度（文本中出现了多少医学专业术语）、图谱概念覆盖率（命中了多少不同的医学概念）。综合评分低于 0.15 的文档直接 rejected（不入库），0.15-0.35 的标记为 unverified（入库但 source_type=unverified，在排序中权威分仅 0.3）。

第五层是检索排序的天然压制——即使 unverified 文档侥幸入库，它的 low source_authority_score (0.3 vs guideline 的 1.0) 使其很难在融合排序中进入 top 5。第六层是前端可标记警告——管理页面对 linked_node_codes 为空的文档显示「可能为无关文档」标签。

这不是「一条防线拦死所有」，而是纵深防御——越可疑的文档在排序中越下沉，即使入库也很难被检索出来。

### Q8：你的 Agent 如何保证医疗场景的安全边界？

**A**：五层防线。

**第一层——规则引擎**：体检指标的阈值判断由确定性规则（JSON 配置）完成，不是 LLM 判断。比如「空腹血糖 ≥ 7.0 mmol/L → FBG_diabetes」「肌酐男 ≥ 110 / 女 ≥ 90 → CREATININE_high」。规则不会因为 LLM 的 prompt 不同而改变判断。

**第二层——图谱推理**：风险的疾病映射和干预建议走图谱的关系路径，不是 LLM 自由联想。图谱的节点和边是手工整理+医学指南校验的，不会凭空产生不存在的疾病关联。

**第三层——Workflow 强制执行**：初诊路径是 12 节点确定性流水线，LLM 只能拿到最终结构化结果做自然语言格式化。LLM 不能在中途「觉得这个指标不严重」而跳过某步——Workflow 的节点顺序由代码编排，不可被 LLM 绕过。

**第四层——Agent 约束**：ReAct Agent 的 system prompt 和工具描述双重约束——工具描述中明确「当用户提供体检指标时必须优先调用评估工具」「严禁凭空捏造诊断结果」。Agent 的 `_decide_next_action()` 方法中，有记忆时优先用它而不是调工具，减少 LLM 自由发挥的空间。

**第五层——人工复核标记**：诊断格式化时，只要存在 high 级别风险、或有禁忌提示、或风险数量 > 1，自动设置 `human_review_required=True`。前端会展示「建议结合医生进一步复核，不应替代线下诊疗」的提示。

---

### Q9：系统做了哪些性能优化？效果量化吗？

**A**：分四个层面。

**检索层**：Milvus native distance 复用（消除 Python 侧全量 embedding 点积）、SQLite FTS5 替代内存 BM25（持久化 + 大数量级性能）、MMR 候选裁剪到 15（避免 O(n²) 两两 cosine）、Rerank 前置（让最强的语义信号主导排序方向）。综合效果：全链路 MRR 相对 baseline 提升 27.5%。

**流水线层**：条件短路——无异常指标时跳过图谱检索、证据检索、排序三个最重的步骤，健康路径延迟从 ~350ms 降至 ~50ms（下降 86%）。图谱未命中时跳过干预路径扩展。

**API 层**：图谱风险检索和证据检索无数据依赖，通过 `asyncio.gather` 并行执行。墙钟时间从 ~350ms 降至 ~220ms（下降 37%）。Neo4j driver 改为 AsyncGraphDatabase，Rerank 改为 httpx AsyncClient。

**记忆层**：摘要 LLM 由每轮触发改为累计字符阈值 2000 触发，减少 LLM 调用频率；事实记忆做向量化语义检索，上下文构建时只注入相关 top 5 而非全量。

---

### Q10：36 项测试覆盖了什么？你在测试上投入的思路是什么？

**A**：覆盖了系统五条核心链路和三条横切关注点。

核心链路：(1) 输入解析→指标归一化（test_input_parser, test_indicator_normalizer）(2) 规则判定→单指标+组合规则+性别分层（test_rules_config）(3) 检索排序→Milvus distance 复用、FTS5 rebuild/search、MMR 裁剪、Rerank 前置（test_evidence_retrieval, test_retrieval_round2, test_round3_final）(4) Agent→ReAct 循环决策、记忆优先/工具调用分支、流式事件类型完整性（test_react_agent, test_medical_agent）(5) 会话记忆→诊断版本化、趋势 diff、摘要阈值触发、冲突检测（test_chat_history_service, test_round3_final）。

横切关注点：(1) 同步/异步一致性——`run_state()` 和 `run_state_async()` 产出相同结果 (2) 配置驱动验证——新增规则不需要改代码 (3) 降级验证——Neo4j/Milvus 不可用时自动回落 InMemory。

测试策略上，我的思路是：不追求用例数量，而是**每条核心链路有一个「端到端」的验证测试**（从输入到输出），再对链路中**最容易出 bug 的节点单独测试**（如规则判定中的 AND/OR 嵌套、检索排序中的 ablations、记忆中的冲突检测）。这样既保证整体链路不退化，也保证关键节点的边界条件被覆盖。

---

### Q11：如果给你两周时间继续迭代，你会做什么？

**A**：三个方向，按优先级。

**优先级最高——LLM 真 streaming**：当前 Agent 的回答是生成完再 chunk 推送。改为 LLM token-level streaming（`stream=True` + async iterator），前端可以像 ChatGPT 一样逐字看到回答生成。这需要在 ReAct Agent 的 final_answer 阶段改为 streaming 输出，技术上不复杂但对用户体验是质变。

**第二——构建标准评测集**：手工标注 100+ 体检场景的 ground-truth 检索结果，形成可复现的检索评测 pipeline。这比当前的 50 条评测集更稳固，且可以纳入 CI/CD——每次改检索链路跑一次评测看指标有没有退化。

**第三——Agent 工具细化**：当前 ReAct Agent 只有一个工具 `lookup_medical_knowledge`。拆成 4 个细粒度工具——`check_user_facts`（查事实记忆）、`check_last_diagnosis`（查最近诊断）、`calculate_trend`（对比两次诊断变化）、`lookup_medical_knowledge`（查知识库证据）。工具细化后 Agent 的决策更透明，面试时也更能体现 Agent 工程能力。

**医疗 KAG Agent 工作台 — 基于知识图谱与混合检索的体检辅助评估系统**

### 项目简介

构建面向体检场景的智能辅助评估 Agent 系统，采用「确定性 Workflow + 自研 ReAct Agent」双层架构。用户提交自然语言体检文本后，系统通过 12 节点 LangGraph 流水线完成指标解析、阈值规则判定、Neo4j 图谱多跳风险推理、混合证据检索（Milvus 向量 + SQLite FTS5 词汇 + Rerank 语义重排 + MMR 多样性）与排序融合，产出结构化诊断评估和干预建议。在多轮追问中，ReAct Agent 循环接管交互，基于四层记忆（事实 / 诊断版本化 / 对话 / 摘要）和知识库检索工具灵活作答。

系统具备完整的知识库管理链路、医学相关性门控防污染、三层持久化后端、SSE 流式交互、组件级健康检查与自动降级。经 50 场景标准评测集验证，检索 MRR 0.80、Precision@5 0.79、nDCG@5 0.75，全链路相对纯向量 baseline 提升 27.5%。36 项回归测试覆盖全链路。

---

### 主要工作

**1. Agent 架构设计与 ReAct 循环**

- 设计并实现了「LangGraph 确定性流水线 + 自研 ReAct Agent」双层架构。底层 12 节点流水线作为不可绕过的安全执行引擎，上层 ReAct Agent 通过分层决策策略（记忆优先 → 工具检索 → 综合回答）处理多轮追问。两条路径在 `stream_assess()` 中自动分流：初诊强制执行 Workflow，追问走 ReAct Agent 结构化事件流
- ReAct 循环内置 5 轮迭代上限、工具参数 hash 循环检测（连续重复即强制终止）、7 种结构化流式事件（thinking → decision → tool_call → tool_result → warning → synthesizing → final_answer），全链路对前端可见，无 LangChain 版本耦合
- 医疗安全五层防线：规则引擎（确定性阈值判定）→ 图谱推理（结构化关系路径）→ Workflow 强制编排（LLM 不可绕过）→ Agent 约束（禁止捏造诊断）→ 人工复核标记（high risk 自动标注）

**2. 混合检索排序与评测**

- 设计多查询规划器（最多 8 维度查询）→ 稠密向量（Milvus HNSW native distance 复用） + 词汇检索（SQLite FTS5 BM25 持久化索引，中文 fallback LIKE）双路 RRF 召回 → Rerank 前置语义重排（权重 45%）→ 图谱节点重叠度 + 来源权威度 + 词汇匹配分融合 → 质量门控（graph_overlap > 0 或 score ≥ 0.75）→ MMR 多样性去重（λ=0.75, top 15 候选）。全链路 5 步递进
- Ablation study 覆盖 5 种配置对照，验证各模块收益：Rerank 贡献 +11% MRR，图谱信号贡献 +9% MRR，全链路相对纯 dense baseline 提升 +27.5% MRR
- 最终检索得分 MRR 0.80 / Precision@5 0.79 / nDCG@5 0.75

**3. 多层记忆与上下文工程**

- PostgreSQL + SQLAlchemy 四层记忆模型：(a) 用户事实记忆（含新旧冲突检测，embedding → Milvus 语义检索只召回相关 top 5，防止 prompt 膨胀）(b) 诊断记忆（版本化 is_current + version_no，追问注入最近两版做趋势 diff）(c) 对话记忆（原文 + 摘要双字段，上下文窗口自动裁剪）(d) LLM 摘要记忆（累计字符阈值 2000 触发，避免每轮调用 LLM）
- 上下文超出处理：`_clip_total_history()` 从新到旧累积，新消息优先保留；三级长度控制（单条 500 字符 / 最近 12 条 / 总 3200 字符）

**4. 知识图谱设计与反污染**

- Neo4j 医学图谱：IndicatorState → DiseaseRisk → Disease → Intervention/Department/FollowUp/Contraindication，Cypher 多跳查询，AbstracBase 封装双后端（Neo4j / InMemory）
- 六层防污染：文件类型白名单 → content_hash 去重 → 内容抽取判空 → MedicalRelevanceGate（图谱节点链接数 + 医学关键词密度 + 概念覆盖率三维评分，<0.15 直接 rejected）→ 来源权威度降权（unverified=0.3 vs guideline=1.0）→ 前端标签告警

**5. 性能优化与韧性**

- 异步并行：图谱检索 + 证据检索无依赖并行（asyncio.gather），墙钟时间下降 37%
- 条件短路：无异常指标跳过图谱/证据/排序三步（延迟从 ~350ms → ~50ms，降 86%）
- Neo4j/Milvus 不可达自动降级 InMemory，降级原因 transparent 暴露
- 全链路 12+ 降级点结构化日志，组件级健康检查（6 个后端状态 + 延迟检测）
- 36 项 pytest 回归测试
