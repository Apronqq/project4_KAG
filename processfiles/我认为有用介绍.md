## 问题一：检索相关性如何保证？做过哪些优化？准确率如何？

### 1.1 相关性保障机制（多层递进）

检索相关性在项目中通过**四个层级**逐层保障：

**第一层：查询构建层（Query Planning）**

不是简单地把用户原话扔给检索引擎。`EvidenceQueryPlanner.build_queries()` 会根据上下文生成多维度查询：

- 用户原始问题 → `user_question` 查询
- 前 3 个高风险疾病 → `risk_guideline` 查询（如「高血压 成人 体检 指南 风险识别 干预建议」）
- 前 4 个异常指标 → `abnormal_indicator` 查询（如「收缩压 显著升高 风险 分层 指南」）
- 所有异常指标的聚合总结 → `indicator_summary` 查询

每个查询独立召回、各自贡献 RRF 分数，多查询的结果在 RRF 框架下融合，避免单查询视角偏差。

**第二层：双路召回层（Dense + Lexical）**

- **稠密向量召回**：Milvus HNSW 索引，直接用 native distance 作为分数，不经过 Python 侧二次计算
- **词汇检索召回**：SQLite FTS5 持久化索引，BM25 排序。FTS5 无法匹配中文时自动回退到 LIKE 字符级匹配

两路通过 RRF（Reciprocal Rank Fusion, k=60）融合，各自贡献 50% 的 RRF 累积分。

**第三层：多信号重排序层**

最终分数由五路信号加权融合（`finalize_scores`）：

| 信号           | 权重 | 含义                                                         |
| -------------- | :--: | ------------------------------------------------------------ |
| Rerank 语义分  | 45%  | 远程 Rerank 模型对 query-document 语义相关性的深度判断       |
| 图谱节点重叠分 | 20%  | chunk 的 linked_node_codes 与当前图谱推理路径的 state/risk/disease code 交集比例 |
| 词汇匹配分     | 15%  | FTS5 BM25 归一化分数                                         |
| 稠密向量分     | 10%  | Milvus HNSW cosine distance 归一化                           |
| 来源权威分     | 10%  | 指南=1.0, 药品标签=0.9, 诊疗方案=0.85, 一般文献=0.75         |

**第四层：质量门控与去重层**

- **相关性门控**：`_passes_relevance_gate()` — 图谱重叠分 > 0 或 final_score ≥ 0.75 才允许通过
- **MMR 多样性选择**：λ=0.75，在相关性（75%）和多样性（25%）间平衡，避免返回全部相似 chunk
- **Rerank 前置**：语义重排放在 fusion 之前（占 45% 权重），而不是放在最后做微调。这保证了最强大的语义信号主导排序方向，结构化偏置只是修正

### 1.2 做过哪些优化（按时间顺序）

1. **Milvus distance 复用**：消除 Python 侧对全量 chunk 的 embedding 点积二次计算，直接复用 Milvus search 返回的 distance
2. **BM25 → SQLite FTS5**：从 Python 内存 BM25 迁移到持久化 FTS5，支持大数据量且重启不丢索引
3. **MMR 候选裁剪**：从全量 O(n²) 两两 cosine → 截断到 15 候选再 MMR
4. **Rerank 前置 + 权重调整**：Rerank 从最后 25% 权重提升到最前 45% 权重，调序能力质变
5. **多查询 RRF 融合**：从单查询检索改为多维度并行查询 + RRF 框架，提升召回覆盖面

### 1.3 准确率数据

坦诚地说，项目目前是原型系统阶段，还没有构建标准测试集做正式的 MRR/nDCG/Precision@K 评测。但我在架构上预留了评测接口：

**如果构建评测集，我会这样设计：**

- **测试集构建**：手工标注 50 个体检场景（从 fixtures/exam_cases 抽样 + 增量），每个场景标注 3-5 个 ground-truth 相关 chunk_id
- **评测指标**：MRR（Mean Reciprocal Rank，衡量第一个相关结果的排名位置）、Precision@5（前 5 结果中相关占比）、nDCG@5（考虑排序质量的归一化折损累计增益）
- **对照实验设计**：依次关闭 Rerank / 图谱信号 / 词汇检索 / MMR，做 ablation study

**基于行业经验和我对检索链设计的信心，预期效果**：

- 纯向量检索 baseline：MRR ≈ 0.55-0.60, P@5 ≈ 0.50-0.55
- 加入词汇检索（RRF 双路）：MRR ≈ 0.65-0.70, P@5 ≈ 0.60-0.65
- 加入图谱信号 + 来源权威：MRR ≈ 0.72-0.78, P@5 ≈ 0.68-0.73
- 全链路（含 Rerank 前置 + MMR）：MRR ≈ 0.78-0.85, P@5 ≈ 0.75-0.82

理由：Rerank 前置在公开 benchmark（如 BEIR）上通常带来 8-12% 的 nDCG 提升，图谱重叠信号对医疗领域的结构化查询有额外的精准匹配优势，MMR 虽然可能略微降低 P@5 但显著提升用户感知的多样性。

**实际验证方式**：

我会先在 10 个典型体检场景上做人工标注，跑全链路 vs ablation 对照，确认各模块收益方向和量级，然后迭代权重。如果 Rerank 收益不达预期（<5%），会回退权重分配；如果图谱信号在某些场景误引导（chunk 有匹配 node 但内容不相关），会降低图谱权重或加入「图谱节点匹配数下限」阈值。

## 问题二：会话记忆消息保存策略？如何确保有效数据保留？上下文超出时如何处理？

### 2.1 记忆分层保存策略

不把所有对话一股脑塞进 prompt。项目设计了**四层记忆**，每层有独立的保存策略和生命周期：

| 记忆层                                | 存储位置                  | 保存内容                                                     | 生命周期                                                     |
| ------------------------------------- | ------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **用户事实记忆** (`UserFactMemory`)   | PostgreSQL                | 性别、年龄、每项体检指标值+单位、病史、用药、过敏史          | 会话级，每次新体检数据全量覆盖写入，旧事实做冲突检测后提示更新 |
| **诊断记忆** (`DiagnosticMemory`)     | PostgreSQL                | 每次评估的健康状态、紧急程度、风险摘要、异常指标、科室/复查/生活方式/用药建议、证据摘要 | 会话级，新诊断 version_no+1，旧标记 is_current=false，保留最近 N 版 |
| **对话记忆** (`ConversationMemory`)   | PostgreSQL                | 每轮 user/assistant 原始消息 + content_summary 摘要版        | 会话级，持久化全部消息，但上下文构建时只取最近 12 条         |
| **摘要记忆** (`conversation_summary`) | PostgreSQL ChatSession 表 | LLM 生成的长期对话摘要                                       | 累计新增对话字符数 ≥ 2000 时才触发 LLM 重新生成，未达阈值用确定性摘要 |

### 2.2 有效数据保留的五个策略

**策略一：事实冲突检测而非静默覆盖**



```python
# upsert_user_fact_memory 的核心逻辑
# 1. 先查旧事实 → 2. 删旧写新 → 3. 逐条对比 → 4. 有变化显式标记
conflicts = _detect_fact_conflict(existing_map, "exam_item", 
    "fasting_blood_glucose", "7.2", "mmol/L")
# 输出: "已更新事实：fasting_blood_glucose 从 6.1mmol/L 更新为 7.2mmol/L"
```

冲突信息作为 system 消息注入对话，前端以黄色 banner 展示。用户知道「系统已识别到你这次数据与上次不同」，而不是悄悄覆盖。

**策略二：诊断版本化而非覆写**

每次新诊断不删除旧诊断，而是 `is_current=false` + `version_no+1`。追问时注入最近两版做 diff：



```
"最新诊断(v2)：健康状态=高风险，紧急程度=高
 上次诊断(v1)：健康状态=亚健康，紧急程度=低
 对比变化：收缩压从142升至176 mmHg，新增肌酐升高"
```

这支撑了趋势类追问（「我比上次严重了吗」），而不是每次都从零开始。

**策略三：消息双字段（原文 + 摘要）**

`ConversationMemory` 存两份：`content`（完整原文）+ `content_summary`（截断到 500 字符）。上下文构建时优先用摘要版，非初诊追问时用截断版。这样既保留了完整消息用于历史回放，又控制了上下文长度。

**策略四：结构化摘要而非全量消息**

`record_assistant_message` 时，如果带了 `structured_result`（诊断 JSON），summary 会是结构化字段拼接：



```
"健康状态=high_risk；紧急程度=high；主要风险=高血压风险(high),慢性肾病风险(high)；建议科室=肾内科,心内科"
```

这个结构化摘要直接进入 `conversation_summary` 字段作为长期记忆，比保留 LLM 生成的 300 字自然语言回答更精准。

**策略五：事实记忆的嵌入语义检索**

用户事实记忆做 embedding 存入专用 Milvus collection。追问时不做全量注入，而是用当前问题做向量检索，只召回相关 top 5 事实。例如用户问「我血糖怎么样了」，只召回 `fasting_blood_glucose=7.2` 和 `hba1c=6.8`，不把 15 条指标全塞进去。

### 2.3 上下文超出时的处理

当 `build_context()` 构建的总字符数超过 `CHAT_TOTAL_CONTEXT_CHAR_LIMIT`（默认 3200 字符）时，`_clip_total_history()` 执行以下策略：



```python
def _clip_total_history(self, history, total_limit):
    # 反转列表 → 从新到旧累积 → 超出限制就停止
    clipped = []
    total = 0
    for item in reversed(history):
        if total + len(item["content"]) > total_limit and clipped:
            continue  # 不追加，但继续尝试后面的（可能更短）
        clipped.append(item)
        total += len(item["content"])
    return list(reversed(clipped))  # 恢复时间顺序
```

特点：**保留新消息优先**。先倒序处理，新消息先进 `clipped`。当某条消息加上后会超限制且已有消息时，跳过该条。这保证：

- system prompt 中的事实记忆和诊断记忆一定被保留（它们在最前面，倒序时最后处理，不会被截掉）
- 最近的对话轮次优先保留
- 较早但短的消息仍有可能被保留（长消息被跳过后继续尝试后面的短消息）

此外还有**三级长度控制**：

| 级别       | 机制                             |   阈值    |
| ---------- | -------------------------------- | :-------: |
| 单条消息   | `content_summary` 截断           | 500 字符  |
| 近期消息量 | `chat_recent_messages_limit * 2` |   12 条   |
| 总上下文   | `_clip_total_history`            | 3200 字符 |

## 问题三：知识图谱如何保证有效？如何防止无关文件污染？

### 3.1 图谱有效性保障

**Schema 层**：Neo4j 启用了唯一性约束（`CONSTRAINT indicator_state_code`、`CONSTRAINT disease_risk_code`、`CONSTRAINT disease_code`），防止重复节点。每次启动时 `ensure_schema()` 幂等执行。

**数据层**：图谱数据来源有三条路径，各有限制：

1. **Seed 数据**（`STATE_TO_RISK` + `DISEASE_TO_INTERVENTIONS`）：手工整理的医学图谱，指标→风险→疾病→干预的关系链经过人工校验，是图谱的骨架
2. **知识库文档**：外部上传的 PDF/指南等，不直接写入图谱节点，而是以 `EvidenceChunk` 方式通过 `NODE_LINKED_CHUNK` 边关联到已有风险/疾病节点，不会创建新的节点类型或改变图谱结构
3. **自动推断**：`_infer_linked_node_codes()` 对上传文档做关键词匹配，只有命中已知的 INDICATOR_ALIASES、STATE_TO_RISK、DISEASE_TO_INTERVENTIONS 中的 code/名称时，才建立链接

**推理层**：图谱检索（`get_risk_candidates`）只查有约束的节点类型（IndicatorState → DiseaseRisk → Disease），不会走「不相关」的路径。

### 3.2 防止无关文件污染 —— 六层防线

**第一层：文件类型白名单**



```python
# routes/medical.py
supported_suffixes = {".txt", ".md", ".pdf", ".json", ".html", ".htm"}
```

不在此列表中的文件（如 `.exe`、`.docx`、图片、视频）在上传阶段直接拒绝 400。

**第二层：内容哈希去重**

`content_hash = hashlib.sha256(content).hexdigest()` 持久化在 `KnowledgeDocumentRegistry` 中。上传前先查 hash，命中则返回 `status=duplicate`，跳过所有处理。即使文件名不同、来源不同，只要内容完全一致就不会重复入库。

**第三层：内容抽取验证**

文档解析后得到纯文本。如果文本为空（PDF 是扫描图片、HTML 全是脚本标签），`_split_to_chunks()` 返回空列表 → `chunk_count=0` → 不上传任何 chunk。

**第四层：图谱节点链接推断门控**

`_infer_linked_node_codes()` 只匹配**已知医学概念**。如果一个文件讲的是「如何养宠物」，全文遍历后 `linked_node_codes=[]`——没有命中任何 INDICATOR_ALIASES、STATE_TO_RISK、DISEASE_TO_INTERVENTIONS 的关键词。这意味着：

- 该文件的 chunk 仍然会被存入 Milvus（文本检索层面可用）
- 但图谱重叠分 `graph_overlap_score` 始终为 0
- 在相关性门控 `_passes_relevance_gate` 中，它的 `final_score` 必须 ≥ 0.75 才能通过
- 由于没有图谱信号 + 来源权威分默认 0.7，它很难超过 0.75 的门槛
- 即使侥幸通过门控，没有 `graph_overlap_score` 加持的 chunk 在最终排序中也处于劣势

**第五层：来源权威度分级（已实现但可增强）**

当前 `source_authority_score` 按 source_type 分四档。如果进一步细化，可以在上传时对文档做**自动医学相关性打分**：



```
相关性评分 = f(医学关键词命中数, linked_node_codes 数量, 文档长度)
```

低于阈值的文档标记为 `source_type="unverified"`, `source_authority_score=0.3`，远低于指南的 1.0，自然在检索排序中被压制。

**第六层：人工审核标记（可在前端增强）**

知识库文档列表已展示文件名、类型、chunk 数、关联节点数。可以为 `linked_node_codes=[]` 的文档增加前端警告标签「未关联任何医学图谱节点，可能为无关文档」，并提供一键删除入口。

### 3.3 如果面试官追问「真的有人传了无关文件怎么办」

诚实回答：当前系统对无关文件**不是零容忍而是降权压制**——它不会被拒绝入库，但会在检索排序的各环节被逐步降权，最终很难出现在用户可见的 top 5 结果中。如果要做零容忍，最简单的方案是在上传时增加一个 **医学相关性检查**：如果 `len(linked_node_codes) < 3`，返回 `status=rejected` 并提示「该文档未检测到足够的医学相关内容，请确认后再上传」。这个改动只需在 `_process_upload_job` 的 `chunk_document` 之后加一个 if 判断，约 10 行代码。
